# TripAI Eval

## 1. 目的

`src/eval/` 用于承载 TripAI `8B Domain SFT` 阶段的评测规范、评测脚本和评测产物。

当前评测的目标不是验证 tool calling、function calling 或旧链路 `intentionName` JSON，
而是验证模型是否已经成为一个可用的旅游领域自然语言助手。

本目录服务于以下目标：

1. 建设可复现、可比较、可冻结的评测闭环
2. 为第一轮 `stage1 baseline` 提供统一版本判定标准
3. 为后续第二轮、第三轮数据迭代提供稳定对照
4. 及时发现数据质量问题、风格漂移、能力退化和 bad case 回归

---

## 2. 当前阶段边界

本目录中的评测规范严格服从仓库当前主线：

- 当前主任务是 `8B Domain SFT`
- 当前目标是训练旅游领域自然语言能力
- 当前模型不是工具路由器
- 当前阶段不评测 tool calling / function calling
- 当前阶段不要求输出旧链路 JSON
- 当前阶段不以替代完整线上业务链路为目标

评测时必须默认以下前提：

1. 只评测 `system` / `user` / `assistant` 三类自然语言消息
2. 不把 `tool_calls`、`function_call`、`observation` 等轨迹纳入评测主集
3. 不以是否生成结构化 JSON 作为通过标准
4. 不要求模型回答实时不可确认的价格、库存、票务、开放时间等高时效字段

---

## 3. 训练集、验证集、评测集的区别

这三个概念必须严格区分：

### 3.1 训练集

用于模型训练的主数据。

当前仓库中的第一轮正式训练集目标是六类任务混合后的 `4000 ~ 6000` 条高质量样本，
推荐以 `5000` 条作为 `v0 / stage1 baseline train set` 的目标量级。

### 3.2 训练期验证集

训练框架内部使用的小比例验证集，例如：

- [llamafactory_stage1_sft.yaml](d:\soulv_llm\configs\llamafactory_stage1_sft.yaml) 中的 `val_size: 0.02`
- 训练过程中的 `eval_loss`

它的作用是帮助训练过程选最优 checkpoint，
但它 **不是** 产品版本评审时使用的正式业务评测集。

### 3.3 正式评测集

正式评测集用于：

1. 横向比较不同训练轮次
2. 冻结 `baseline`
3. 判断是否进入下一轮数据建设
4. 判断是否允许部署、演示或对外汇报

正式评测集必须：

- 与训练集严格隔离
- 不参与训练
- 版本固定后尽量不频繁改动
- 保留独立的评分规则和评测报告

---

## 4. 第一轮评测集建设原则

第一轮正式训练数据接近收齐时，就应该同步开始建设评测集，
而不是等所有训练轮次结束后再补评测。

推荐顺序：

1. 六类任务 schema 基本稳定
2. 第一轮训练集接近 `5K`
3. 同步抽取并人工编写独立评测样本
4. 冻结 `stage1 baseline eval set`
5. 基于该评测集训练并评审第一轮 baseline 模型

### 4.1 推荐规模

第一轮正式评测集建议总量：

- `300 ~ 600` 条

推荐起步方案：

- 六类主评测集：`50 ~ 80` 条 / 类
- 总量约：`360 ~ 480` 条

此外建议额外维护一份回归集：

- `50 ~ 150` 条 bad case

回归集与主评测集用途不同：

- 主评测集：做版本横向比较
- 回归集：盯历史失败案例是否再次出现

### 4.2 六类推荐分布

如果主评测集总量在 `420` 条左右，可参考如下分配：

| task_type | 建议条数 | 说明 |
| --- | ---: | --- |
| `guide_generation` | 90 | 长文本、结构化生成，波动最大 |
| `travel_qa` | 90 | 高频问答，覆盖广 |
| `hotel_recommendation` | 70 | 需求理解和推荐理由都要看 |
| `traffic_planning` | 60 | 可执行性重要 |
| `persona_understanding` | 50 | 重点看因人而异 |
| `multi_turn_dialogue` | 60 | 重点看上下文继承与改口 |

如果评测资源有限，也可以先做每类 `50` 条的最小版，
先保证六类都被覆盖，再逐步扩充。

## 5. 评测样本来源要求

评测样本必须尽量独立于训练主料，避免“训练过的题换个说法再评测”的假提升。

推荐来源：

1. 人工编写的全新问题
2. 从真实业务问题中抽样后重新改写
3. 从 bad case 归纳出的高价值边界问题
4. 不同城市、不同人群、不同预算、不同天数的组合题

不建议直接作为正式评测样本的来源：

1. 直接从训练集复制
2. 只改几个字的近重复样本
3. 模板痕迹很重、答案唯一性很弱的样本
4. 强依赖实时价格、班次、库存、票务的样本

---

## 6. 当前阶段的评测重点

当前阶段评测应聚焦“旅游领域自然语言能力”，主要检查以下方面：

1. 是否真的回答了用户问题
2. 是否符合对应任务类型的输出风格
3. 是否保持 TripAI 旅游助手的自然口吻
4. 是否理解预算、时间、同行人群、偏好、限制条件
5. 是否避免编造高时效信息
6. 是否在多轮里正确继承和更新上下文
7. 是否比上一轮更稳定、更少模板化复读

---

## 7. 评测维度

建议每条样本按以下公共维度评分：

| 维度 | 含义 |
| --- | --- |
| `correctness` | 是否真正回答到点上，事实和常识是否基本正确 |
| `instruction_following` | 是否遵守用户约束和 system persona |
| `completeness` | 该说的是否说到，是否遗漏关键条件 |
| `clarity` | 结构是否清楚、表达是否自然 |
| `safety_and_honesty` | 不确定信息是否谨慎表达，是否避免瞎编 |
| `brand_style` | 是否符合 TripAI 轻品牌、自然、不过度营销的风格 |

推荐使用 `1 ~ 5` 分制：

- `5`：明显优秀，可直接接受
- `4`：基本满意，只有轻微问题
- `3`：可用但存在明显缺陷
- `2`：问题较大，不建议放行
- `1`：明显失败

### 7.1 通用否决项

出现以下任一情况时，该样本可直接判为失败：

1. 明显答非所问
2. 多轮对话中忽略关键约束更新
3. 虚构具体价格、库存、票务、开放时间且表达为确定事实
4. 输出旧链路 JSON、tool trace 或与任务无关的结构化内容
5. 明显营销堆砌、品牌导流过强，盖过问题本身
6. 内容空泛，几乎没有可执行信息

## 8. 六类任务专项评测要求

### 8.1 `guide_generation`

重点检查：

1. 是否有清晰的行程或内容结构
2. 是否围绕用户给定条件生成，而不是套模板
3. 是否避免整篇堆砌景点介绍
4. 是否给出有用的节奏、路线、注意事项或实操建议
5. 品牌口吻是否自然，不过度导流

高风险失败：

1. 开头高度模板化
2. 所有城市都写成同一种结构
3. 只会堆景点，不会规划
4. 把实时价格和开放信息当作确定事实写死

### 8.2 `travel_qa`

重点检查：

1. 回答是否简洁直接
2. 是否真正解决用户问的景点、城市、交通知识问题
3. 不确定的实时信息是否表达为一般性建议而非硬事实
4. 是否避免无关的长篇攻略化输出

高风险失败：

1. 问答题写成营销长文
2. 明明不知道却给出确定答案
3. 漏答核心问题

### 8.3 `hotel_recommendation`

重点检查：

1. 是否识别预算、人群、位置、偏好、交通便利度等约束
2. 是否给出“推荐对象 - 推荐理由”对应关系
3. 是否体现不同需求下推荐差异
4. 是否避免编造具体库存、房态和实时价格

高风险失败：

1. 只列酒店，不讲为什么适合
2. 忽略预算或同行人群
3. 推荐逻辑千篇一律

### 8.4 `traffic_planning`

重点检查：

1. 路线建议是否可执行
2. 是否考虑出发地、目的地、时间、换乘复杂度
3. 是否提供一般性交通建议，而不是乱写实时班次票价
4. 是否在不确定时明确说明需要二次确认

高风险失败：

1. 虚构班次、票价、剩余票量
2. 建议不可执行
3. 忽略用户的时间或出发地点条件

### 8.5 `persona_understanding`

重点检查：

1. 是否真的理解用户画像
2. 是否体现“因人而异”的建议
3. 是否把预算、偏好、同行人群转化为不同方案
4. 是否避免套话式“都适合去看看”

高风险失败：

1. 不同画像给出几乎同样的答案
2. 只复述用户条件，不做推理
3. 推荐结果和用户画像冲突

### 8.6 `multi_turn_dialogue`

重点检查：

1. 是否正确继承上下文
2. 是否对改口、补充、覆盖、约束更新做出响应
3. 是否知道哪些旧条件应保留，哪些已失效
4. 是否避免每轮都从头重写整篇长文

高风险失败：

1. 忽略最新约束
2. 保留已经被用户推翻的旧条件
3. 多轮状态混乱

## 9. 评测集文件格式建议

当前阶段建议评测样本也沿用 ChatML 风格，
但比训练样本多保留一些评测元信息。

推荐字段如下：

```json
[
  {
    "id": "eval_guide_000001",
    "split": "golden",
    "task_type": "guide_generation",
    "scene": "city_travel",
    "difficulty": "medium",
    "tags": ["city_trip", "3_days", "family"],
    "messages": [
      {
        "role": "system",
        "content": "系统提示词"
      },
      {
        "role": "user",
        "content": "用户问题"
      }
    ],
    "reference_answer": "人工参考答案，可长可短，不要求唯一。",
    "must_include": [
      "围绕亲子出行",
      "体现三天节奏"
    ],
    "must_not_do": [
      "编造实时票价",
      "输出工具调用"
    ],
    "notes": "用于评估亲子三日游攻略生成。"
  }
]
```

字段说明：

| 字段 | 是否必需 | 说明 |
| --- | --- | --- |
| `id` | 必需 | 样本唯一 ID |
| `split` | 必需 | 建议值：`golden` / `regression` / `shadow` |
| `task_type` | 必需 | 六类任务之一 |
| `messages` | 必需 | 待评测输入，至少包含 `user`，可保留 `system` |
| `reference_answer` | 建议 | 参考答案，不要求唯一，但要体现好答案方向 |
| `must_include` | 建议 | 关键点清单 |
| `must_not_do` | 建议 | 常见错误清单 |
| `difficulty` | 建议 | `easy` / `medium` / `hard` |
| `tags` | 建议 | 便于切片统计 |
| `notes` | 可选 | 标注人员备注 |

要求：

1. `messages` 中当前阶段只允许 `system`、`user`、`assistant`
2. 正式待评测输入通常只需要 `system + user`，不应把参考答案放进模型输入
3. `reference_answer` 用于人工或自动判分参考，不是训练标签
4. `must_include` 与 `must_not_do` 应尽量具体，避免空泛

## 10. 评测集切分建议

建议至少维护以下三类切分：

### 10.1 `golden`

主评测集。

用途：

1. 比较 `baseline` 与后续版本
2. 产出对外可汇报的稳定指标
3. 作为是否进入下一轮训练的主要依据

要求：

- 一经确认，尽量冻结
- 非必要不改
- 修改必须记录版本变更原因

### 10.2 `regression`

历史 bad case 回归集。

用途：

1. 看某轮优化是否修复历史问题
2. 防止修了一个问题又引入旧问题

要求：

- 可持续增长
- 每个 bad case 最好带 issue 来源或失败说明

### 10.3 `shadow`

候选样本池。

用途：

1. 暂存还未冻结的新评测题
2. 观察是否适合作为下轮 `golden` 补充

要求：

- 不直接参与正式版本对比
- 经审核后再升格到 `golden`

## 11. 评分方式建议

当前仓库尚未落地完整自动评测脚本，
因此推荐采用“规则校验 + 人工评测 + 回归检查”三层结合。

### 11.1 第一层：规则校验

适合自动检查：

1. 输出是否为空
2. 是否误输出 JSON / tool trace
3. 是否明显过短
4. 多轮格式是否完整
5. 是否触发 `must_not_do` 中的硬性问题

### 11.2 第二层：人工评分

人工评分应作为当前阶段的主判定方式。

建议每条样本至少记录：

1. 总分
2. 公共维度分
3. 是否通过
4. 失败原因标签
5. 备注

### 11.3 第三层：回归检查

对 `regression` 集单独统计：

1. 修复率
2. 回退率
3. 新增失败类型

---

## 12. 第一轮 baseline 的建议判定口径

第一轮 `stage1 baseline` 不必追求极限分数，
但必须至少满足“六类都可用、整体风格稳定、无明显致命硬伤”。

建议从以下角度看结果：

1. 六类任务是否都有基本可用结果
2. 是否存在大量模板复读
3. 是否经常编造高时效信息
4. 是否在多轮对话中频繁忘记约束
5. 是否比 bootstrap 版本更稳定

在第一轮阶段，不建议只看一个总分。
至少应同时保留：

1. 总体通过率
2. 六类分项通过率
3. 六类平均分
4. 关键 bad case 修复情况

## 14. 评测样本编写规范

编写评测题时，必须避免“题目看起来像业务，实际只是在测背模板”。

推荐做法：

1. 多写真实用户口吻，而不是规范化题干
2. 同一任务覆盖不同城市、不同天数、不同预算、不同人群
3. 保留少量模糊输入，看模型是否会合理澄清或给出稳妥建议
4. 保留少量边界题，看模型是否知道谨慎表达

不推荐做法：

1. 所有题都写得过于教科书
2. 每类只写最标准、最容易的一种问法
3. 全是知识问答，没有约束条件题
4. 全是长文本，没有短问短答

## 15. 评测结果报告最少应包含什么

每次正式评测，报告最少应包含：

1. 模型名称或 checkpoint 标识
2. 评测集版本
3. 评测时间
4. 总样本数
5. 六类样本数
6. 总体通过率
7. 六类通过率
8. 六类平均分
9. 失败样本示例
10. 与上一版本对比结论

如果只输出一个总分，不足以支撑数据迭代决策。

## 16. 当前仓库落地建议

基于当前仓库状态，建议下一步按以下顺序推进：

1. 先冻结第一轮六类正式训练集
2. 同步建设 `golden` 主评测集和 `regression` 回归集
3. 训练 `stage1 baseline`
4. 产出第一版正式评测报告
5. 再决定第二轮是续训小实验还是累计数据重训

注意：

- 当前训练配置里的 `val_size` 只能替代训练期验证，不能替代正式业务评测
- 当前 `src/eval/` 仍处于规范先行阶段，本 README 是后续评测脚本和评测数据建设的统一依据

## 17. 一句话原则

当前阶段的评测，不是在测模型会不会“走工具”，
而是在测它能不能稳定地做一个自然、靠谱、不过度乱编的 TripAI 旅游助手。
