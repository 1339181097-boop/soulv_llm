# TripAI 32B Stage1 SFT 六类数据需求文档

版本：`v2.0`  
适用阶段：`Qwen3-32B stage1`  
当前阶段用途：给数据部门/数据员做抽取、整理、交付的正式口径单

## 1. 这次要收什么

本轮只收 `stage1` 六类自然语言 SFT 数据：

1. `guide_generation`
2. `travel_qa`
3. `hotel_recommendation`
4. `traffic_planning`
5. `persona_understanding`
6. `multi_turn_dialogue`

这轮模型要学会的是：

- 写攻略
- 回答旅游问答
- 理解住宿、交通、人群、预算、偏好等需求
- 根据用户补充条件调整建议
- 在多轮里继承上下文并改口

这轮不学：

- 意图分类标签
- JSON 路由结果
- `tool_calls`
- 工具执行轨迹
- 下单、查询、办理类结果

## 2. 总量目标与分配

### 2.1 最终清洗可用量

`32B stage1` 默认按 `10K` 组织，推荐最终清洗可用量如下：

- `guide_generation: 650`
- `travel_qa: 3250`
- `hotel_recommendation: 1900`
- `traffic_planning: 2000`
- `persona_understanding: 1300`
- `multi_turn_dialogue: 900`

说明：

- 最终进入训练前仍会做 token-aware mix
- `guide_generation` 和 `multi_turn_dialogue` 不允许为了凑量盲目堆长文模板
- 当前 8B 最弱的三类是 `traffic_planning`、`persona_understanding`、`multi_turn_dialogue`
- 这三类优先追求推理逻辑、约束处理、上下文继承，不是只追求条数

### 2.2 原始候选申请量

因为后面还要清洗、去重、过滤低质量内容，原始候选量必须高于最终保留量。

建议原始候选交付量：

- `guide_generation: 3000-5000`
- `travel_qa: 8000-12000`
- `hotel_recommendation: 4500-6500`
- `traffic_planning: 3500-5000`
- `persona_understanding: 3000-4500`
- `multi_turn_dialogue: 1800-2500` 组会话

### 2.3 交付批次

建议按三批交付：

1. 第 1 批：每类 `30-50` 条，做字段/格式 smoke
2. 第 2 批：先交约 `30%` 正式候选，供清洗与 stage1 小跑
3. 第 3 批：补齐剩余正式候选

## 3. 统一交付要求

### 3.1 文件格式

- 统一交付 `jsonl`
- 编码 `UTF-8`
- 一行一条记录
- 不要 BOM
- 不要 Excel 导出乱码

### 3.2 文件命名建议

- `guide_generation_raw.jsonl`
- `travel_qa_raw.jsonl`
- `hotel_recommendation_raw.jsonl`
- `traffic_planning_raw.jsonl`
- `persona_understanding_raw.jsonl`
- `multi_turn_dialogue_raw.jsonl`

### 3.3 通用最小字段

所有类别尽量至少保留：

```json
{
  "record_id": "唯一记录ID",
  "task_type": "guide_generation",
  "source": "tripai_db / 客服日志 / 内容库 / 人工整理",
  "source_id": "原系统主键或原文ID",
  "city": "城市名，若适用",
  "destination": "目的地，若适用",
  "title": "标题，若适用",
  "updated_at": "原始内容更新时间，若有",
  "tags": ["亲子", "预算型", "美食"]
}
```

### 3.4 推荐补充字段

如果能提供，尽量多给：

- `district`
- `days`
- `scene`
- `audience`
- `budget_level`
- `content`
- `user_query`
- `assistant_content`
- `note`
- `is_time_sensitive`
- `people_count`
- `travel_style`

### 3.5 统一不收什么

这轮不要收：

- 实时班次
- 实时票价
- 实时余票
- 实时库存房态
- 下单/预订/支付结果
- 工具调用轨迹
- 旧链路意图识别结果
- 纯 JSON 路由输出

## 4. 通用质量要求

### 4.1 优先保留

- 内容完整
- 结构清晰
- 语言自然
- 信息密度高
- 有真实旅游帮助价值
- 能体现用户需求差异

### 4.2 优先剔除

- 乱码
- 明显截断
- 全文几乎一样的模板复读
- 纯广告导流
- 大量堆砌关键词
- 强实时但没有时间标记的内容
- 明显过时、无来源、无法判断真假的内容

### 4.3 去重原则

重点去重：

- 同一城市、同一天数、同一开头、同一章节结构
- 同一问题换少量同义词但答案几乎一样
- 同一酒店、景点、路线被重复导出多次

## 5. 六类详细要求

### 5.1 guide_generation

一句话定义：  
用户提出目的地、主题或行程需求，系统给出一份结构完整、可读性强、可直接参考的旅游攻略或分日安排。

重点要：

- 有明确目的地
- 有行程结构
- 最好有天数或分段安排
- 有交通、美食、准备事项、tips 等实用信息
- 能看出是在帮用户规划，不是在给搜索引擎堆料

不要：

- 空泛欢迎词
- 纯广告页
- 只有景点列表没有安排逻辑
- 票价、营业时间、导流按钮占主体
- 完全复制粘贴的模板文案

数量口径：

- 最终目标 `650`
- 原始候选 `3000-5000`

### 5.2 travel_qa

一句话定义：  
用户问一个旅游问题，回答应直接、自然、清楚，不需要写成长攻略。

重点要：

- 景点、城市、交通、住哪里更方便等常见问答
- 能在 `1-3` 段内说清
- 回答简洁直接
- 不确定的实时信息用稳妥话术表达

不要：

- 纯实时票价问答
- 纯实时营业时间问答
- 纯政策查询结果
- 问答写成营销长文

数量口径：

- 最终目标 `3250`
- 原始候选 `8000-12000`

### 5.3 hotel_recommendation

一句话定义：  
不是让模型订酒店，而是让模型理解住宿需求，并解释为什么推荐某类区域、类型或酒店特征。

重点要：

- 用户需求文本
- 住宿推荐理由
- 区域优缺点
- 不同人群、预算、位置、出行目的下的差异化建议

不要：

- 实时房价
- 库存房态
- 下单、预订、支付结果
- 纯销售型文案

数量口径：

- 最终目标 `1900`
- 原始候选 `4500-6500`

### 5.4 traffic_planning

一句话定义：  
给用户可执行的通用交通建议，而不是返回实时班次和票价。

重点要：

- 某地到某地常见怎么去
- 哪种方式更省时间、更省心、更适合老人孩子
- 景点串联顺不顺路
- 分天行程里的交通安排建议

不要：

- 实时班次
- 实时票价
- 实时余票
- 必须在线查才能成立的结果

数量口径：

- 最终目标 `2000`
- 原始候选 `3500-5000`

### 5.5 persona_understanding

一句话定义：  
同一个目的地，因为用户人群、预算、偏好不同，推荐方式也要明显不同。

强结构要求：

- `persona_type` 必填
- `people_count` 必填
- `preference_tags` 必填
- `avoid_tags` 必填
- `budget_level` 必填
- `candidate_spots` 必填且至少 `3` 个，推荐 `5` 个
- 每个 `candidate_spot` 至少带：
  - `name`
  - `tags`
  - `brief`
  - `price`

重点要：

- 用户是谁
- 用户在意什么
- 用户不想要什么
- 推荐为什么会因人而异
- 同城不同画像的差异化建议

不要：

- 所有人群都给同一套答案
- 只有标签没有自然语言解释
- 只有画像标签罗列没有推荐逻辑

数量口径：

- 最终目标 `1300`
- 原始候选 `3000-4500`

### 5.6 multi_turn_dialogue

一句话定义：  
用户在对话中不断补充条件、改口、否定前文、更新约束，模型需要跟着调整回答。

强结构要求：

- 必须提供完整 `messages` 或等价 `turns`
- 角色严格交替
- 至少 `3` 组 `user/assistant`
- 鼓励提供 `constraint_changes`

重点要：

- 有上下文承接
- 用户新增约束后，回答会变化
- 用户否定前文后，回答会修正
- 不同轮次信息会累积

禁止：

- “其余不变”“同上续写”“后续同上”这类伪多轮
- 两轮纯闲聊拼接
- 完全没有承接关系的问答拼接
- 每轮都重写一整篇超长攻略

数量口径：

- 最终目标 `900`
- 原始候选 `1800-2500` 组会话

## 6. 数据员提交前自查

请至少自查：

1. 是否分成 `6` 个文件
2. `task_type` 是否正确
3. 是否有乱码、空内容、明显截断
4. 是否混入意图识别或工具调用数据
5. 是否混入大量实时票价、库存、班次结果
6. 是否保留了 `source`、`source_id`、`updated_at` 等可追溯字段

## 7. 给数据员的一句话判断规则

如果一条数据的核心价值是：

- 帮用户写攻略
- 回答旅游问题
- 理解住宿、交通、人群、预算需求
- 根据补充条件修改回答

那大概率可以收。

如果一条数据的核心价值是：

- 判意图
- 查实时结果
- 返回 JSON
- 下单、查询、办理
- 记录工具轨迹

那这轮不要收。
