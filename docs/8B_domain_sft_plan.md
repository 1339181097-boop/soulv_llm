# TripAI 本地模型训练方案（8B阶段）
版本：v1.0  
目标模型：Qwen3-8B（当前阶段）  
后续正式基座：Qwen3-32B-Instruct  
适用阶段：Domain SFT（旅游领域能力打底阶段）

---

## 1. 当前阶段目标

### 1.1 总目标
基于 Qwen3-8B，完成一个可训练、可评测、可部署的旅游领域基础模型，作为后续 tool-use / function calling / 32B 增强的基础。

### 1.2 当前阶段要解决的问题
当前阶段不以“替代旧业务链路”为目标，也不要求模型沿用旧的“意图识别 JSON 输出形式”。  
当前阶段主要解决：

1. 旅游领域语言能力打底
2. 旅游攻略生成能力
3. 景点/城市/交通知识问答能力
4. 酒店/交通/出行需求理解能力
5. 用户偏好和基础多轮能力
6. 公司数据库内容的可利用能力
7. 本地训练、评测、部署全流程跑通

### 1.3 当前阶段不强求
以下内容不是 8B Domain SFT 阶段的主目标：

1. tool calling / function calling
2. agent 编排
3. DPO / RLHF
4. 复杂业务替代
5. 完整线上链路迁移

---

## 2. 与旧方案的关系

### 2.1 旧方案中可保留的部分
旧技术方案中以下方向可保留：

- 分阶段后训练思路
- 阶段一：领域 SFT
- 阶段二：工具调用训练
- 评测集建设
- RAG / 向量检索
- 旅游结构化知识资产沉淀

### 2.2 旧方案中需要调整的部分
当前阶段不再以旧链路的“意图识别 JSON”作为主任务。  
改为先做 **Domain SFT**，后续再进入 **Tool-use SFT**。

### 2.3 当前阶段的正确定位
当前 8B 阶段应理解为：

> 先把通用模型训练成“TripAI 风格的旅游领域基础模型”，让它具备攻略生成、旅游问答、需求理解、偏好多轮等能力；再在后续阶段单独训练工具调用能力。

## 4. 8B 阶段数据总体策略

### 4.1 总体原则
数据不是越多越好。  
当前阶段优先级：

1. 数据质量
2. 任务划分清晰
3. 模板控制
4. 可评测
5. 可复现

### 4.2 第一轮推荐数据规模
第一轮（v0）推荐总量：

- **4000 ~ 6000 条高质量样本**

说明：
- 这个量已经足够跑通第一轮训练、评测、推理
- 不需要一上来就上 2 万条以上
- 第一次训练的重点是验证方案，不是追求最终上限

### 4.3 第二轮推荐数据规模
第二轮（v1）可扩到：

- **8000 ~ 12000 条**

### 4.4 第三轮以后再考虑扩容
当 v0 / v1 跑通后，再考虑补到：

- **15000+**
- 甚至更高

但不建议第一轮就追求大规模堆料。

## 5. 8B SFT 数据组成

### 5.1 数据桶划分
当前阶段 SFT 数据按 6 类组织：

1. 攻略生成（guide_generation）
2. 景点 / 城市 / 交通问答（travel_qa）
3. 酒店 / 住宿推荐与需求理解（hotel_recommendation）
4. 交通规划（traffic_planning）
5. 用户画像 / 偏好理解（persona_understanding）
6. 多轮对话（multi_turn_dialogue）

### 5.2 推荐配比（v0）
第一轮不要继续只按“样本条数占比”理解六类数据，而要同时约束 token 占比。

原因：
- `guide_generation` 和 `multi_turn_dialogue` 天然更长。
- `travel_qa` 虽然条数可以很多，但单条通常很短。
- 如果继续只按 sample 数量混合，长文本桶会在训练里被明显放大。

因此 v0 不建议：
- 直接照搬旧的条数比例。
- 只看 bucket 样本数。
- 使用纯 weighted sample mixing。

建议采用 token-aware 配比：
- `guide_generation`：800
- `travel_qa`：1000
- `hotel_recommendation`：750
- `traffic_planning`：500
- `persona_understanding`：500
- `multi_turn_dialogue`：500

说明：
- 如果机械按 `1500/1250/750/500/500/500` 配，长文本桶的训练权重会明显失衡。
- 以当前数据长度分布估算，`guide_generation` 的 token 占比很容易超过 50%，甚至接近 60%。
- 因此需要把六类目标改成 token 更均衡的版本，而不是继续沿用按条数拍脑袋分配的方案。

### 5.3 当前可执行的融合方案（v0）
正式六类版：
- `guide_generation`：800
- `travel_qa`：1000
- `hotel_recommendation`：750
- `traffic_planning`：500
- `persona_understanding`：500
- `multi_turn_dialogue`：500
- 总计：4050

若 `persona_understanding` 暂时还没有 processed 数据，则先落五类候选版：
- `guide_generation`：800
- `travel_qa`：1000
- `hotel_recommendation`：750
- `traffic_planning`：500
- `multi_turn_dialogue`：500
- 总计：3550

执行原则：
- v0 阶段优先使用 stage mixer 的 `filename=count` 精确配额方式。
- 不再使用只看 sample weight 的旧式 stage 配比。
- `travel_qa` 进入混合前必须先做严格去重与答案限频。
- `guide_generation` 进入混合前必须先做噪声过滤与 token 控制。

## 6. 公司数据库如何利用

### 6.1 数据分层原则
公司数据库里的数据不要“全量直接喂训练”，而要分成三层使用：

#### A. 直接用于 SFT 的数据
适合作为训练主料：
- 城市攻略
- 主题攻略
- 景点介绍（高质量）
- 酒店推荐文案
- 交通建议
- 常见问答
- 偏好型推荐样本

#### B. 优先用于 RAG / 检索的数据
更适合做检索增强，而不是直接做主训练料：
- 长篇景点详情
- 长篇酒店详情
- 变化频繁的信息（票价、开放时间、营业时间、实时政策）
- 大量重复模板介绍

#### C. 优先用于索引/标准化的数据
不适合直接做主训练样本，但很适合后续能力增强：
- 城市名 / 景点名 / 酒店名
- 别名 / 简称 / 错别字映射
- 坐标 / 行政区 / 区域层级
- 标签（亲子、情侣、拍照、美食、预算、夜景等）

## 7. 模板化严重的攻略数据如何处理

### 7.1 结论
数据库里这类模板化攻略数据 **可以使用**，但不能原样全量硬喂。

### 7.2 为什么不能直接全量使用
这类数据通常有以下问题：

1. 开头模板重复严重
2. 章节结构完全相同
3. 口吻过于固定
4. 品牌导流句高频重复
5. 内容存在冗余堆料
6. 时效信息可能过期
7. 模型容易学成“模板复读机”

### 7.3 TripAI 品牌内容是否保留
**保留。**

因为 TripAI 是自有品牌，品牌身份属于产品 persona 的一部分。  
但要做“控制性保留”，不是“所有样本全都强品牌化”。

---

## 8. 模板攻略清洗规范（重点）

### 8.1 清洗总原则
保留：
- 旅游内容结构
- 品牌身份
- 核心知识点
- TripAI 的风格特征

控制：
- 强模板重复
- 高重复营销话术
- 固定欢迎词
- 无差别广告导流

---

### 8.2 清洗目标
让模型学会：

1. TripAI 的品牌身份
2. 攻略的完整结构
3. 自然的旅游表达
4. 按用户需求调整内容

而不是学会：

1. 每次都同样开场
2. 每次都先打一段广告
3. 每次都强营销
4. 所有城市都生成同一种模板

---

### 8.3 模板攻略数据的三种版本

#### A. 强品牌版
特点：
- 保留 TripAI / 小奇身份
- 保留欢迎词
- 可保留少量导流内容
- 风格更活泼

适用：
- 教模型产品风格和 persona

建议占比：
- **20% ~ 30%**

---

#### B. 轻品牌版（主力）
特点：
- 保留 TripAI 身份
- 保留轻度品牌口吻
- 降低导流强度
- 内容优先

适用：
- 作为 SFT 主力数据

建议占比：
- **50% ~ 60%**

---

#### C. 内容优先版
特点：
- 品牌露出更轻
- 基本不导流
- 更强调内容组织
- 更适合问答、局部推荐、多轮修改

适用：
- 让模型避免过度模板化

建议占比：
- **20% ~ 30%**

---

### 8.4 模板攻略清洗操作规范

#### 必做 1：去重
规则：
- 同一城市、同一天数、同一章节结构、同一开头高度相似的样本，不全收
- 高重复文本只保留代表样本
- 重复度过高的样本要去重或只留少数版本

#### 必做 2：开头多样化
问题：
- “嘿，旅行者……太棒了……让我这个旅游界的小奇带你……”如果大量重复，会让模型只会一种开头

要求：
- 同一类攻略至少准备多种开场风格
- 允许保留一部分原始强风格，但不能全量统一

#### 必做 3：导流句降频
问题：
- 如果每条样本都出现“打开 TripAI App / 点击商城购票”，模型会被训成广告机

要求：
- 导流句只在部分样本中保留
- 不同样本中导流句的位置、强度要不同
- 问答类样本、局部推荐样本尽量少导流

#### 必做 4：拆长文
一条完整长攻略，不应只作为一条样本。  
需要拆成多个训练样本，例如：
- 城市一日游攻略
- 景点推荐
- 交通建议
- 美食推荐
- tips
- 局部行程规划
- 轻量问答
- 多轮修改

#### 必做 5：控制时效信息
以下信息不要过度依赖 SFT 硬记：
- 门票价格
- 营业时间
- 预约入口
- 打车价格
- 实时活动

处理方式：
- 可保留部分常识性内容
- 高频变化信息后续优先交给 RAG / tool-use

#### 必做 6：减少堆料
问题：
- 长攻略中常有大量重复信息、过多列表、冗余描述

要求：
- 保留重点知识点
- 保留结构
- 去掉明显堆料和重复表述

## 9. 各类数据的清洗要求与规范

### 9.1 攻略生成数据（guide_generation）

#### 保留内容
- 标题结构
- 出行准备
- 交通建议
- 美食推荐
- 行程安排
- tips
- 天气提示
- 特产推荐
- TripAI 品牌身份

#### 清洗要求
1. 开头不能全部一致
2. 品牌可保留，但导流要降频
3. 行程内容应尽量贴合城市特点
4. 避免明显重复段落
5. 长文要拆分成短任务样本
6. 强模板样本只保留部分原样版本

#### 质量要求
- 内容完整
- 结构稳定
- 风格自然
- 不过度广告化
- 不全部同模板

---

### 9.2 景点 / 城市 / 交通问答（travel_qa）

#### 来源
- 景点详情文本
- 城市介绍
- 交通信息
- 结构化字段

#### 清洗要求
1. 问法多样化
2. 答案简洁清楚
3. 减少固定营销话术
4. 避免每个问答都套攻略口吻
5. 对高时效字段做标记

#### 不建议
- 所有问答都以“嘿，旅行者”开头
- 所有问答都插品牌广告

#### 质量要求
- 信息明确
- 表达自然
- 适合直接回答用户

---

### 9.3 酒店 / 住宿推荐与需求理解（hotel_recommendation）

#### 来源
- 酒店结构化信息
- 酒店标签
- 用户需求模板
- 推荐文案

#### 清洗要求
1. 重点保留“需求 -> 推荐理由”关系
2. 不要求接工具
3. 不要大量虚构价格和库存
4. 适度保留品牌风格，减少硬广
5. 强化预算 / 人群 / 位置 / 偏好 这些约束词

#### 质量要求
- 能体现需求理解
- 推荐理由清楚
- 语言自然
- 不过度销售化

---

### 9.4 交通规划（traffic_planning）

#### 来源
- 城市交通建议
- 机场/高铁站到市区
- 景点间交通建议
- 日程中转移建议

#### 清洗要求
1. 优先保留通用建议
2. 高频变化票价/班次不要过度依赖
3. 回答要可执行
4. 减少营销导向

#### 质量要求
- 实用
- 简洁
- 逻辑清晰

---

### 9.5 用户画像 / 偏好理解（persona_understanding）

#### 来源
- 用户偏好标签
- 亲子 / 情侣 / 老人 / 学生党 / 预算型 / 特种兵模板
- 约束型推荐数据

#### 清洗要求
1. 明确保留用户特征
2. 输出应体现“因人而异”的推荐方式
3. 同一城市做不同人群版本
4. 减少所有样本同一种话术

#### 质量要求
- 能体现个性化
- 不同画像有明显区分

---

### 9.6 多轮对话（multi_turn_dialogue）

#### 来源
- 从单轮攻略改造
- 从推荐类样本改造
- 人工构造改口 / 补充条件 / 否定前文

#### 清洗要求
1. 多轮里要体现状态变化
2. 要体现修改、覆盖、补充
3. 不要只是简单拼接两轮闲聊
4. 每轮要有明确意图变化或约束更新

#### 质量要求
- 有上下文承接
- 能体现条件更新
- 回答不是每轮都重写整篇长文

---
