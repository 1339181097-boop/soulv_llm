# Persona Understanding 数据生成规范

## 1. 目标

本文档用于规范 `persona_understanding` 类原始数据的生成方式，目标是：

- 保证数据真实、可控、可追溯
- 避免依赖大模型自由生成导致的虚构内容
- 保证样本符合 SFT 六类数据中“用户画像理解”任务要求

核心原则：

1. 先基于数据库规则筛选候选景点
2. 再用固定模板组织自然语言
3. 不允许补充数据库中不存在的事实

---

## 2. 数据来源约束

`persona_understanding` 数据只能基于现有景点数据库构造。

当前允许使用的数据字段包括但不限于：

- `city`
- `spot_name`
- `price`
- `address`
- `open_hours`
- `play_index`
- `tags`

如某字段不存在，则禁止在输出中编写对应事实。

禁止写入以下高风险内容，除非数据库中有明确字段支撑：

- 具体公交/地铁线路
- 步行时间、车程时间、排队时长
- Wi-Fi、会议空间、商务接待能力
- 无人机飞行条件
- 餐厅、咖啡店、美妆店、Spa 等真实商户信息
- 实时开放情况、实时客流、实时预约情况

---

## 3. 任务定义

`persona_understanding` 的核心不是“编一篇攻略”，而是：

- 识别用户画像
- 理解预算、偏好、限制条件
- 从候选景点中筛出更匹配的点
- 用自然语言说明“为什么这些景点更适合这个人”

因此输出重点应该是“因人而异的推荐逻辑”，而不是大量运营细节。

---

## 4. 推荐原始数据结构

建议 raw 层统一使用 JSONL，每行一条样本，字段如下：

```json
{
  "record_id": "persona_000001",
  "task_type": "persona_understanding",
  "source": "tripai_db",
  "city": "北京",
  "persona_type": "亲子",
  "user_query": "带6岁孩子在北京玩2-3天，想轻松一点，最好互动性强，也别太贵，有什么适合的景点推荐？",
  "audience": ["亲子"],
  "preference_tags": ["互动体验", "轻松节奏", "儿童友好", "预算敏感"],
  "avoid_tags": ["高消费", "台阶多", "纯成人向"],
  "budget_level": "low",
  "people_count": 3,
  "candidate_spots": [
    {
      "name": "景点A",
      "price": "免费",
      "address": "北京市xx区xx路",
      "open_hours": "09:00-17:00",
      "tags": ["亲子", "室内", "互动"]
    },
    {
      "name": "景点B",
      "price": "门市价￥20起",
      "address": "北京市xx区xx街",
      "open_hours": "08:30-17:30",
      "tags": ["儿童友好", "轻松", "科普"]
    }
  ],
  "assistant_content": "如果你更看重亲子互动和轻松节奏，建议优先考虑景点A、景点B和景点C。这几个点要么互动性更强，要么对孩子更友好，而且整体花费相对可控，比较适合带小朋友慢慢玩。",
  "reason_text": "该画像重点偏好互动体验、儿童友好和预算可控，因此优先选择亲子标签明显、价格较低、节奏更轻松的景点。",
  "constraints": ["高消费", "台阶多", "纯成人向"]
}
```

说明：

- `task_type` 固定为 `persona_understanding`
- `constraints` 必须使用正确拼写，不使用 `contraints`
- `candidate_spots` 是事实来源，后续文案中提到的景点必须来自这里
- `assistant_content` 和 `reason_text` 都不能超出 `candidate_spots` 与画像标签所支持的事实范围

---

## 5. 画像类型建议

建议优先做一组固定画像类型，避免无限扩散。

推荐画像类型：

- 亲子
- 情侣
- 老人
- 学生党
- 预算型
- 摄影爱好者
- 独自旅行
- 三代同堂
- 商务出差
- 闺蜜游
- 美食爱好者

每种画像都应包含：

- 人群特征
- 预算水平
- 核心偏好
- 明确避雷项

---

## 6. 景点标签体系

为了避免模型自由发挥，必须先给景点做结构化标签。

建议标签包括：

- 亲子
- 互动体验
- 儿童友好
- 室内
- 免费
- 低价
- 古建
- 夜景
- 拍照
- 观景
- 轻松
- 少步行
- 休息设施
- 文化体验
- 自然风景
- 本地特色

如果某个标签没有人工或规则依据，则不要标。

---

## 7. 生成流程

### 第一步：按城市取 Top30 景点

每个城市从景点表中按“好玩指数”选取前 30 个景点，作为候选池。

### 第二步：为景点补充结构化标签

标签来源可以是：

- 数据库已有标签字段
- 基于已有字段的规则映射
- 人工校对后的标签表

不要依赖大模型直接给景点打标签。

### 第三步：定义画像规则

每种画像需要一套固定筛选逻辑。

例如：

- `预算型`：免费、低价景点加分；高价景点减分
- `亲子`：亲子、互动、室内加分
- `老人`：少步行、轻松、休息设施加分
- `摄影爱好者`：拍照、观景、古建、夜景加分
- `独自旅行`：轻松、本地特色、探索性加分

### 第四步：按规则打分并筛选景点

从城市 Top30 中筛选出最适合该画像的 3 到 5 个景点。

推荐做法：

- 必须命中核心偏好标签
- 尽量避开与 `avoid_tags` 冲突的景点
- 不够匹配时宁可少推荐，不要编造

### 第五步：生成 `user_query`

通过固定模板生成用户问题，只替换城市、人群、偏好、预算等变量。

### 第六步：生成 `assistant_content`

通过固定模板，把“用户画像 + 候选景点 + 匹配原因”组织成自然语言。

### 第七步：生成 `reason_text`

只解释推荐逻辑，不补充任何新事实。

### 第八步：人工抽检

抽检生成数据是否符合规则，重点检查字段一致性和事实边界。

## 8. 画像筛选规则示例

### 8.1 预算型

偏好：

- 免费
- 低价
- 本地特色
- 经济实惠

避雷：

- 高消费
- 昂贵
- 不必要的开销

加分规则示例：

- `免费`：+3
- `低价`：+2
- `本地特色`：+2
- `文化体验`：+1

减分规则示例：

- 高价景点：-3
- 标签与预算冲突：-2

### 8.2 亲子

偏好：

- 互动体验
- 儿童友好
- 轻松节奏
- 室内或安全性高

避雷：

- 高强度活动
- 台阶多
- 纯成人向

加分规则示例：

- `亲子`：+3
- `互动体验`：+3
- `儿童友好`：+2
- `室内`：+1

### 8.3 老人

偏好：

- 少步行
- 轻松
- 休息设施
- 经典景点

避雷：

- 陡峭台阶
- 高强度活动
- 长时间步行

加分规则示例：

- `少步行`：+3
- `轻松`：+2
- `休息设施`：+2
- `文化体验`：+1

### 8.4 摄影爱好者

偏好：

- 拍照
- 夜景
- 古建
- 观景

避雷：

- 无特色
- 不出片
- 人潮过密

加分规则示例：

- `拍照`：+3
- `观景`：+2
- `夜景`：+2
- `古建`：+2

## 9. 文案模板

## 9.1 `user_query` 模板

模板 1：

```text
+作为{persona_type}，想在{city}玩2-3天，更看重{preference_1}和{preference_2}，预算{budget_desc}，有什么更适合我的景点推荐？
```

模板 2：

```text
+我准备去{city}玩2天左右，属于{persona_type}，希望整体偏{preference_1}、{preference_2}，同时尽量避开{avoid_1}，怎么选景点更合适？
```

模板 3：

```text
+如果我是{persona_type}，在{city}旅游时更关注{preference_1}、{preference_2}和{preference_3}，有哪些景点会更匹配？
```

## 9.2 `assistant_content` 模板

模板 1：

```text
+如果你更看重{preference_1}和{preference_2}，建议优先考虑{spot_1}、{spot_2}、{spot_3}。这几个点分别更适合{reason_1}、{reason_2}，而且整体更符合你对{budget_or_style}的要求。
```

模板 2：

```text
+从用户画像来看，你更适合选择{spot_1}、{spot_2}这一类景点，而不是偏{avoid_desc}的景点。前者在{preference_1}、{preference_2}上更匹配，整体体验也会更贴合你的出行需求。
```

模板 3：

```text
+如果按{persona_type}的需求来选，推荐重点关注{spot_1}、{spot_2}、{spot_3}。它们在{preference_1}、{preference_2}和{preference_3}上更有优势，也更容易避开{avoid_1}、{avoid_2}这类问题。
```

## 9.3 `reason_text` 模板

模板 1：

```text
+该画像的核心需求是{preference_list}，同时希望避免{avoid_list}，因此优先选择标签更匹配、价格更合适、游玩压力更低的景点。
```

模板 2：

```text
+该推荐主要依据用户的人群特征、预算水平和偏好标签生成，重点保留了与{preference_1}、{preference_2}相关的景点，并剔除了与{avoid_1}冲突的候选项。
```

## 10. 严格禁止项

以下内容禁止出现在 `assistant_content` 或 `reason_text` 中，除非数据库明确具备对应字段：

- 地铁几号线、公交几路
- 步行几分钟、车程几分钟
- 实时排队情况
- Wi-Fi 稳定
- 商务接待能力
- 快速通道、预约容易
- 无人机可以飞
- 具体餐厅、咖啡店、美妆店、Spa 店名
- 今日营业情况

原则：

- 没有数据库事实支撑，就不要写
- 宁可简化，也不要补充“看起来合理”的细节

---

## 11. 质量检查清单

生成后必须检查：

1. `task_type` 是否固定为 `persona_understanding`
2. `constraints` 是否拼写正确
3. `people_count` 是否与画像一致
4. `assistant_content` 中提到的景点是否全部来自 `candidate_spots`
5. 是否写入了数据库中不存在的事实
6. 是否混入真实餐厅或商户信息
7. 同一城市下不同画像是否有明显区分
8. 是否只复述用户条件而没有体现推荐逻辑

---

## 12. 最终执行建议

实际落地时，建议按以下方式做：

1. 先整理城市 Top30 景点数据
2. 补齐景点标签表
3. 固定画像类型和画像模板
4. 用规则脚本为每个画像筛景点
5. 用模板生成 `user_query`、`assistant_content`、`reason_text`
6. 做人工抽检和规则校验

一句话总结：

不要让大模型“生成事实”，而要让规则先选出真实景点，再用模板把推荐逻辑写出来。
