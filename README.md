# TripAI 8B MVP 全链路工程上下文

## 一、项目当前定位

当前仓库的主目标，是基于 `Qwen3-8B` 跑通一个旅游领域基础模型的 **8B MVP 全链路**：

1. 数据清洗与构造
2. 本地训练
3. 本地评测
4. 推理验证
5. 可部署闭环

当前活跃阶段是 **8B Domain SFT**。  
这一阶段的定位，严格以 [docs/8B_domain_sft_plan.md](/d:/soulv_llm/docs/8B_domain_sft_plan.md) 为准：

- 先把通用模型训练成 TripAI 风格的旅游领域基础模型
- 重点建设攻略生成、旅游问答、需求理解、偏好多轮能力
- 跑通训练、评测、部署全流程

当前阶段 **不是** 旧业务链路的 `intentionName` JSON 替代方案，也 **不是** tool calling / function calling 阶段。

## 二、当前阶段边界

### 当前阶段做什么

- 训练旅游领域自然语言能力
- 训练攻略生成能力
- 训练景点 / 城市 / 交通基础问答能力
- 训练酒店 / 住宿推荐与需求理解能力
- 训练交通规划能力
- 训练用户画像 / 偏好理解能力
- 训练基础多轮补充、改口、约束更新能力
- 建设可训练、可评测、可部署的 8B MVP 全链路

### 当前阶段不强求什么

- 不强求输出旧链路的意图识别 JSON
- 不做 tool calling / function calling
- 不做 agent 编排
- 不做 DPO / RLHF
- 不做复杂业务执行替代
- 不做完整线上链路迁移

## 三、8B MVP 全链路目标

当前阶段的正确目标是完成一个可落地的旅游领域基础模型闭环，而不是训练“工具路由器”。

当前 8B 模型必须优先具备：

1. 城市 / 区域 / 主题攻略生成
2. 景点 / 城市 / 交通基础问答
3. 酒店 / 住宿需求理解
4. 交通出行建议
5. 用户画像与偏好理解
6. 基础多轮补充 / 改口 / 约束更新
7. TripAI 品牌风格与旅游口吻

## 四、数据总体策略

### 第一轮推荐规模（v0）

- `4000 ~ 6000` 条高质量样本

当前阶段优先级：

1. 数据质量
2. 任务划分清晰
3. 模板控制
4. 可评测
5. 可复现

第一轮重点是验证方案和跑通闭环，不建议一开始就追求大规模堆料。

### 后续规模

- v1：`8000 ~ 12000` 条
- v2+：`15000+`

## 五、8B SFT 数据组成

当前阶段 SFT 数据按 6 类组织：

1. `guide_generation`：攻略生成
2. `travel_qa`：景点 / 城市 / 交通问答
3. `hotel_recommendation`：酒店 / 住宿推荐与需求理解
4. `traffic_planning`：交通规划
5. `persona_understanding`：用户画像 / 偏好理解
6. `multi_turn_dialogue`：多轮对话

### v0 推荐配比

- 攻略生成：`30%`
- 景点 / 城市 / 交通问答：`25%`
- 酒店 / 住宿推荐与需求理解：`15%`
- 交通规划：`10%`
- 用户画像 / 偏好理解：`10%`
- 多轮对话：`10%`

### 如果第一轮总量为 5000 条

- 攻略生成：`1500`
- 旅游问答：`1250`
- 酒店需求理解：`750`
- 交通规划：`500`
- 用户画像理解：`500`
- 多轮对话：`500`

当前 `README` 不再采用旧版的 `itinerary 50% / dialogue 30% / roleplay_safety 20%` 配比描述。

## 六、公司数据库使用原则

公司数据库数据不应全量直接喂给 SFT，而应分层使用：

### A. 直接用于 SFT 的数据

- 城市攻略
- 主题攻略
- 景点介绍（高质量）
- 酒店推荐文案
- 交通建议
- 常见问答
- 偏好型推荐样本

### B. 优先用于 RAG / 检索的数据

- 长篇景点详情
- 长篇酒店详情
- 高频变化信息
- 大量重复模板介绍

其中高频变化信息包括：

- 票价
- 开放时间
- 营业时间
- 实时政策

### C. 优先用于索引 / 标准化的数据

- 城市名 / 景点名 / 酒店名
- 别名 / 简称 / 错别字映射
- 坐标 / 行政区 / 区域层级
- 标签体系

## 七、模板攻略数据处理原则

模板化严重的 TripAI 攻略数据 **可以使用**，但不能原样全量硬喂。

主要问题包括：

1. 开头模板重复严重
2. 章节结构完全相同
3. 口吻过于固定
4. 品牌导流句高频重复
5. 内容存在冗余堆料
6. 时效信息可能过期
7. 模型容易学成模板复读机

### 品牌内容保留原则

TripAI 品牌内容 **保留**，但必须“控制性保留”，不能把所有样本都做成强品牌化模板。

### 模板攻略三种版本占比

- 强品牌版：`20% ~ 30%`
- 轻品牌版（主力）：`50% ~ 60%`
- 内容优先版：`20% ~ 30%`

### 清洗操作重点

1. 去重
2. 开头多样化
3. 导流句降频
4. 长文拆分成短任务样本
5. 控制时效信息硬记
6. 减少堆料和重复表述

## 八、各类数据清洗重点

### `guide_generation`

- 保留完整攻略结构
- 保留 TripAI 品牌身份，但导流降频
- 长文拆分为短任务样本
- 避免全量同模板

### `travel_qa`

- 问法多样化
- 答案简洁清楚
- 减少固定营销话术
- 对高时效字段做标记

### `hotel_recommendation`

- 强化“需求 -> 推荐理由”关系
- 不要求接工具
- 不大量虚构价格和库存
- 强化预算 / 人群 / 位置 / 偏好约束

### `traffic_planning`

- 优先保留通用建议
- 高频变化票价 / 班次不做强依赖
- 回答要可执行
- 减少营销导向

### `persona_understanding`

- 明确保留用户特征
- 输出体现因人而异
- 同一城市做不同画像版本
- 减少单一固定话术

### `multi_turn_dialogue`

- 多轮里体现状态变化
- 体现修改、覆盖、补充
- 每轮有明确约束更新
- 不要每轮都重写整篇长文

## 九、样本格式

推荐统一采用 `messages` 格式，每条样本保留必要元信息：

```json
{
  "id": "guide_000001",
  "task_type": "guide_generation",
  "scene": "city_travel",
  "source": "tripai_database",
  "brand_style": "light_brand",
  "difficulty": "medium",
  "messages": [
    {"role": "system", "content": "..."},
    {"role": "user", "content": "..."},
    {"role": "assistant", "content": "..."}
  ]
}
```
